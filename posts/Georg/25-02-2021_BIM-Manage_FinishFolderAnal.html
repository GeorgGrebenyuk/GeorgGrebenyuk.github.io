Верификация файлов участниками в своих Итоговых контрольных точках

Дата публикации: 25-26.02.2021г.

Курс BIM-менеджмент близится к завершению, дата последней контрольной точки установлена на 9 марта. В связи с этим, необходимо было изменить логику проверок прошлых анализаторов данных, отталкиваясь от структуры файлов, приведенной в ТЗ. Примерно за неделю до этого, в конце прошлой недели Александр Евгеньевич поппросил сделать подобное приложение для самих участников курса - чтобы они запускали его на своей стороне и шла проверка имен файлов, а также наличия нужных файлов по папкам в рамках данной "Итоговой".

Я решил сперва реализовать приложение для проверки файлов на стороне участников, а потом, собрав негативный feedback сделать уже для извлечения данных на нашей стороне. На этот раз, в отличие от приватного репозитория AppsForVC, я решил разместить это приложение в публичном репозитории AppsForVC-NN_Open.

В рамках данной задачи я приоткрыл для себя элемент DataGridView в WinForms, а также освоил, хоть и в базовом варианте, язык разметки markdown для оформления репозитория.

Структура и логика приложения

Структура ничем не отличается от устоявшейся схемы - класс Form1 с элементами управления интерфесом, public static class Actions для выполнения непосредственно логики проверок и public static class Technical для промежуточных операций как-то проверка корректности имени файла, проверка Revit-файлов на наличие временных и т.д. (собственно говоря - это 2 функции тут только).

Процесс работы выглядит следующим образом:

  1. Пользователь запускает программу и выбирает каталог "Исходящие" в рамках своего участка. Это нужно во-первых, чтобы получить наименование участка взяв имя родительской категории у данной (и в дальнейшем использовать спарку "квадрат"-"номер участка"), а во вторых - чтобы проверить корректность именования папки контрольной итоговой точки
  2. Полученное строчное значение FBD_OpenOufFolder.SelectedPath передается методу Actions.CreateReport() класса Actions для непосредственной аналитики файлов в папке (что есть), идет проверка только на корректность имён. При этом сам отчет виду небольшого количества данныз записывается в элемент StringBuider (а может, к слову, и во временный файл сразу). Я реализовал логику записи сперва в StringBuider (в память), а потом - во временный файл в корне "Исходящих". Логика приложения должна стремиться к максимальной оптимизации - потому нет смысла хранить в памяти объем данных, если он не нужен для транзакций и можно легко записать его во временный файл.
  3. Метод Actions.CreateReport() перебирает файлы в строго-типизированных папках (то есть исходит из того, что папки названы верно). Если они названы неверно (исключение Directory.Exists()), то в отчет прописывается соответствующая информация "такая папка не найдена". Ранние отчеты проверяли папки как есть, с выводом в отчет сигнатуры о корректности ее наименования. В этот раз - ввиу того, что пользователь может запустить программу на своей стороне неограниченное число раз - всё жестче.
  4. По окончанию перебора файлов в элемент TextBox справа записывается отчет, какие файлы не были обнаружены. Ввиду того, что основная цель курса - это наполнение информационными моделями Нижнего Новгорода, то первый приоритет за файлами АР-RVT, АР-Renga, Генплан-Revit, NWD-Сборка; второй приоритет - за файлами пректа Lumion (которые будут переданы компании Lumion для маркетинговых целей).
  5. По окончанию работы метода Actions.CreateReport() и записи временного файла в рамках действия по нажатию на кнопку "Запуск" запускается анализ файла отчета (временный файл) с пропуском первой строки-заголовков и информация заносится в табличный элемент DataGridView. Предварительно инициируются 3 колонки данных с заголовками, а заполнение строг отведено циклу в рамках foreach(){} перебора временного файла-отчета.
